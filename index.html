android性能优化

发表于 2017-02-14 | 分类于 android性能优化 | | 阅读次数

对于android开发人员来说，随着应用的功能越来越复杂，布局越来越丰富，对于性能优化是创建高质量的与应用程序的基础。

布局优化

1.避免过度重绘：过度绘制会浪费很多CPU,GPU资源。android系统给我们提供了一种这样的检测工具，在开发者选项中的GPU overdraw。对于我们开发人员来说，尽量增大蓝色区域，减少红色区域。
2.优化布局层次：在android中，系统对View绘制时，通过View树遍历进行，如果view树的高度太高，影响测量和绘制的速度。因此降低view树的高度，优化布局。
3.使用include标签重用Layout:android在布局搭建时，很多界面都会存在一些公用的UI，为了减少代码的冗余，可以使用标签来定义一个公用的UI。
公用UI:

include引入公用布局：

4.使用ViewStub实现View的延迟加载：使用跟include一样，但显示view分为两种：1. VISIBLE:通过调用setVisibility()渲染。2. inflate:通过调用inflate()显示这个view。这两种方式都可显示view,而不同点是第二中可以通过findViewById()里面的控件。
5.使用Hierarchy Viewer工：android官方提供的工具，在sdk的tools中即可找到，可以知道每个view所绘制的时长和view树深度。

内存优化：

1.静态变量导致的内存泄漏：在一个Activity中应用了静态变量，当这个Activity销毁时，这个变量变量无法销毁。（这种模式一般不会发生，一般都会避免使用）
2.单例模式导致的内存泄漏：由于单利模式的特点是其生命周期和Application的保持一致，因此Activity对象无法及时释放。
3.属性动画导致的内存泄漏：属性动画有一类有一类无限循环动画，如果在onDestroy()中没有停止的话，那么动画会一直播下去，尽管界面没有展现动画效果，而这个时候Activity的View持有这个动画，而View又持有Activity，最终Activity无法释放。

4.线程优化：一般采用线程池，避免程序中存在大量的Thread线程池可以重用内部的线程，避免了线程的创建和销毁造所带来的性能开销，同时线程池可以控制线程池的最大并发数，避免大量的线程因抢占资源而导致阻塞现象发生。

响应速度的优化：

响应速度的优化核心是避免主线程中做耗时操作，如果主线程中作太多的耗时操作，会导致Activity启动时黑屏现象，甚至出现ANR,而Android规定，Activity如果5秒之内无法响应屏幕触摸事件或键盘输入事件就会出现ANR,而BroadcastReceiver如果10秒之内还未执行完操作也会出现ANR。当出现ANR后，系统会在data/anr目录下创建一个traces.txt文件，通过分析这个文件可以定位出ANR的原因。

在github上搭建自己的博客

发表于 2016-12-01 | 分类于 github | | 阅读次数

1.首先安装git、node;
2.进入Git Bash,输入下面命令行进行安装hexo；

npm install -g hexo;


3.新建文件夹，用于存放你的博客，将git切换到你的文件夹中，初始化本地仓库输入命令：

git init


4.初始化hexo,输入命令

hexo init;


5.安装hexo的依赖包：通过命令
npm install hexo –save
6.当一切安装完成后，在本地搭建自己的博客，通过命令:

hexo g


7.然后执行命令：
hexo s
8.这时，自己在本地已经搭建完成，可以在浏览器中输入：localhost:4000查看结果：出现以下界面表示本地搭建成功；

9.在github注册账户，注册账户后新建一个仓库，命名格式为:

“用户名.github.io”


10.配置github的ssh,配置好ssh，使用命令检测是否打通：

ssh -T git@github.com


11.如果打通，配置_config.yml文件中的deploy：
type: git
repository: github上的仓库地址
branch: master

12.执行下面命令:
1.hexo g;
2.hexo d;

13.如果执行命令hexo d出现错误，执行命令：npm install hexo-deployer-git –save后，在执行

hexo d


及整个博客创建完成。

对于自定义View处理常用的类

发表于 2016-11-08 | 分类于 自定义view | | 阅读次数

TouchSlop:

TouchSlop是系统所能识别出的最小滑动距离，也就是说，当你在屏幕上滑动时，如果滑动的距离小于这个值，则系统认为没有滑动。通过下面代码即可获取:

int slop = ViewConfiguration.get(this).getScaledTouchSlop();

VelocityTracker:

速度追踪，用于追踪手指在滑动过程中的速度，包括水平或垂直方向的速度。使用方法：

  1. public boolean onTouchEvent(MotionEvent event) {
  2. if(mVelocityTracker==null){
  3. mVelocityTracker= VelocityTracker.obtain();
  4. }
  5. mVelocityTracker.addMovement(event);
  6. mVelocityTracker.computeCurrentVelocity(1000);
  7. float velocityX= mVelocityTracker.getXVelocity();
  8. float velocityY = mVelocityTracker.getYVelocity();
  9. System.out.println(“X轴上的速度”+velocityX+”Y轴上的速度”+velocityY);
  10. return super.onTouchEvent(event);
  11. }

    首先，他是在view的onTouchEvent方法中追踪当前点击事件的速度，需要注意的是，在获取速度之前必须要先计算速度，必须先调用computeCurrentVelocity()方法。最后不需要的时候，要释放资源：
      

    1. mVelocityTracker.clear();
      2. 

    2. mVelocityTracker.recycle();

    GestureDetector

    手势检测，用于辅助检测用户的单击、滑动、长按、双击等行为。比如在Activity实现一个手势检测的过程：
    ~首先必须创建一个GestureDetector对象，并实现OnGestureListener接口。接着在onTouchEvent()方法添加如下：~

@Override
    public boolean onTouchEvent(MotionEvent event) {
        return mGestureDetector.onTouchEvent(event);
    }

Scroller

用于实现view的弹性滑动，对于学习scroller前，必须要知道scrollTo和scrollBy方法，scrollTo是一个绝对滑动，而scrollBy时基于当前位置的相对滑动，在源码中，他调用了scrollTo方法。对于他们来说，滑动的是对于view的内容，在水平方向或垂直方向滚动的距离。而对于scrollTo和scrollBy来说，无法实现弹性滑动，所以就需要使用scroller;

scroller的使用步骤：

1.初始化scroller;
2.复写computeScroll()方法，在内部实现滚动相关的业务逻辑,并进行postInvalidate()的调用；
3.调用startScroll()方法，并进行重绘invalidate();
实现ImageView的动画：
核心代码：


public class ImageScrollerView extends LinearLayout {
    private Scroller mScroller;
    private int startX;
    private int duration;
    private boolean flag=true;
    public ImageScrollerView(Context context) {
        super(context);
    }

public ImageScrollerView(Context context, AttributeSet attrs) {
    super(context, attrs);
    mScroller=new Scroller(context);
    duration=3000;
}

@Override
public void computeScroll() {
    if (mScroller.computeScrollOffset()){
        scrollTo(mScroller.getCurrX(),0);
        invalidate();
    }
}
public void beginScroll(){
    if(flag){
        mScroller.startScroll(mScroller.getCurrX(),0,-250,0,duration);
        startX=mScroller.getCurrX();
        flag=false;
    }else {
        mScroller.startScroll(-startX,0,80,0,duration);
        flag=true;
    }
    invalidate();
}

}

activity的介绍

发表于 2016-11-04 | 分类于 activity ， intent-fillter | | 阅读次数

Activity的生命周期

onCreate()—->onStart()—->onPause()—–>onResume()—->onPause()—–>onStop()—->onDestory()

Paused状态

~表示当Activity失去其焦点，被一个新的非全屏的Activity或者一个透明的Activity放在栈顶时，转换为Paused状态，他失去与用户交互的能力，所有的信息成员变量还保持着。但在系统内存极低的情况下，会被系统回收掉。~

Activity重新创建的过程

~如果系统长期处于stop形态，且系统需要更多内存或内存紧张时，系统会回收Activity,此时会将Activity状态通过onSaveInstanceState()方法保存到Bundle对象中，当你重新创建这个Activity时，保存到Bundle对象就会传递到Activity的onRestoreInstanceState()方法与onCreate()方法中。~

Android任务栈

~在Android中，通过栈结构保存整个App的Activity,栈低的元素是整个任务1的发起者，当一个App启动后，如果当前App不存在App的任务栈，系统就会创建一个，这个App启动的所有Activity都将在这个任务栈中被管理。这个栈被称为Task,同时，一个Task中的Activity可以来自不同的App,同一个App的Activity也可能不在同一个Task中。~

Activity的启动模式

1.Standard:
~这种模式启动一个Activity时，每次实例化。~
2.SignleTop:
~这种启动模式启动一个Activity,系统检测当前栈顶的Activity是否是要启动的Activity,如果是，不实例。这种启动模式在聊天界面，每次接收消息后显示的界面；微信的支付界面。~
3.SignleTask:
~如果以这种方式启动一个Activity,他将创建一个新的任务栈。~
4.SignInstance:
~会创建一个新的任务栈，且该任务栈中只存在这一个Activity。如果应用A创建一个MainActivity,且启动模式为此模式，如果B应用也要激活MainActivity,则不需要创建，两者应该共享。~
SignleTask和SignleInstance中使用startActivityForResult()方法来启动另一个ActivityB,这种情况系统直接返回Activity.RESULT_CANCELED,而不再去等待返回。这种情况只能通过Intent来绑定数据。

Intent Flag启动模式

  • Intent.FLAG_ACTIVITY_NEW_TASK:
    使用一个新的Task启动一个Activity.
  • FLAG_ACTIVITY_SIGNLE_TOP:
    跟SignleTop启动模式一样。
  • FLAG_ACTIVITY_CLEAR_TOP:
    使用SignleTask启动一个Activity
  • FLAG_ACTIVITY_NO_HISTORY:
    ~使用该模式启动一个Activity后，该Activity就会消失，不会在任务栈中~
  • FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS:
    该标记的Acyivity不会出现在历史Activity的列表中，有时候我们不希望我们的Activity返回到历史的列表中，我们就采用次标记
    ###IntentFilter的匹配规则
    ~一个Activity可以有多个intent-filter.~
    action的匹配规则：
    ~Intent中的action必须能够和过滤规则中的action匹配，就是说和action的字符串完全一致。在一个过滤规则中有多个action，只要匹配一个intent中的action能够和过滤规则中的任何一个action相同即可匹配。同时需要注意的是，action是区分大小写的。~
    category的匹配规则：
    ~与actionn不同，如果intent中出现category，不管有几个category，他都必须与intent匹配。~
    data的匹配规则：
    ~data的过滤规则与action的相似，但是data的结构不同于action和category。~
    data的结构，有两部分组成，mimeType和URI两部分组成，mimeType是指媒体类型，例：image/jpeg等，URI的结构如下：
    ‹schem>:/‹host>:‹port>/[‹path>|‹pathPrefix>|‹pathPattern>]
    scheme:URI的模式，比如http、file、content等如果URI中没有指定scheme，则整个URI无效。
    Host:URI的主机名，
    port:URI的端口号。
    path、pathPrefix、pathPattern:表示路径消息，path表示完整的路径，pathPattern:表示完整的路径消息，可以含有通配符；pathPrefix:路径的前缀消息。
    URI默认值为file/content.
    *注意，如果intent中指定了完整的data,就必须调用setDataAndType方法，而不能调用setData或setType,这两个方法会相互清除对方的值。
